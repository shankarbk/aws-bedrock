How the agent proves its identity when calling external tools, APIs, and services

1
LLM decides to call an external tool Agent Trigger

During reasoning, the Foundation Model determines it needs to call an external resource — a REST API, a database, a SaaS platform (Salesforce, Jira), or an AWS service (S3, DynamoDB, Lambda).

  • The agent emits a tool-use block specifying the tool name and parameters
  • AgentCore Runtime intercepts this before any outbound call is made
  • The Runtime enforces AgentCore Policy — checking if the agent is allowed to call this tool for this user
2
AgentCore Identity — credential retrieval Identity

AgentCore Identity looks up the correct credential for this tool call. Credentials are never embedded in agent code or prompts. They live in secure stores.

  • AWS resources: The agent's execution IAM Role is assumed via STS — produces short-lived access key + secret + session token (TTL: 1 hour)
  • OAuth 2.0 services (Salesforce, Google, GitHub): AgentCore uses a pre-authorised OAuth refresh token stored in AWS Secrets Manager → exchanges for a fresh access token via the IdP's token endpoint
  • API Key services: Fetches the key from Secrets Manager at runtime — key is never in plaintext in code
  • User-delegated access: If acting on a user's behalf, AgentCore Identity holds a user-consented OAuth token and uses it (OAuth on-behalf-of flow)
3
AgentCore Gateway — tool routing & policy enforcement Gateway

The Gateway acts as a secure proxy between the agent and the external world. It transforms the LLM's tool-use intent into a real authenticated HTTP/AWS API call.

  • Validates that the tool call parameters match the registered schema (prevents injection)
  • Applies natural-language Cedar policies deterministically — e.g. "agent cannot call DELETE on production S3 bucket"
  • Injects the fetched credential into the outbound request headers
  • Logs every outbound call attempt to CloudTrail with agent session ID, tool name, and parameters
4
Request signing / token attachment Token Exchange

The outbound HTTP request is signed or credentialed based on what the target service expects.

  • AWS services (S3, DynamoDB, Bedrock itself): Request is signed with AWS Signature Version 4 (SigV4) using the STS-issued temp credentials
  • OAuth 2.0 services: Authorization: Bearer <access_token> header is attached
  • API Key services: x-api-key: <key> or custom header injected as required
  • mTLS: For services requiring mutual TLS, the agent's client certificate (stored in ACM) is presented during TLS handshake
5
External service validates & processes External Service

The external service (AWS API, SaaS platform, internal API) receives the authenticated request and processes it.

  • AWS: IAM validates SigV4 signature and checks the Role's permission policies against the requested action and resource
  • OAuth service: validates the Bearer token against its JWKS, checks scopes (e.g. read:contacts), and processes the request
  • Returns the result (data, confirmation, error) as JSON/XML over HTTPS back to the Gateway
  • If auth fails → 401/403 is returned; AgentCore Gateway catches it, logs it, and surfaces a tool-error to the LLM for retry or graceful failure
6
Tool result returned to agent — reasoning continues Success

The Gateway passes the external service's response back to the AgentCore Runtime, which injects it as a tool result into the LLM's context for the next reasoning step.

  • Credentials are never exposed in the tool result — only the data payload is passed to the LLM
  • AgentCore Memory can persist relevant results (e.g. retrieved user data) to the session for future turns
  • The LLM uses the result to continue reasoning and produce the final answer for the user
  • Metrics: latency, token usage, and tool call count emitted to CloudWatch

Interview key points — Outbound Authentication

What: Outbound auth is how the agent proves its identity to external services when making tool calls. The core principle is zero secrets in code or prompts — all credentials are fetched at runtime from Secrets Manager or STS. AgentCore Identity manages credential retrieval; AgentCore Gateway enforces policies and injects credentials. For AWS resources, IAM Role assumption via STS produces short-lived temp credentials — preferred over long-lived keys. For SaaS platforms, OAuth 2.0 with refresh tokens is the standard — the agent never holds user passwords. Cedar policies in AgentCore Policy provide deterministic guardrails on which tools the agent can call and on which resources — enforced before the outbound call, not inside the LLM reasoning loop.
