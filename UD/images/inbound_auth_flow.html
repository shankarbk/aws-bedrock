1
Client / Calling Application Actor

An external application (web app, mobile app, another service, or a human user) wants to invoke the AgentCore agent. Before sending the request it must obtain credentials.

  • Authenticates with its own Identity Provider (IdP) — e.g. Amazon Cognito, Okta, Azure AD, or corporate SSO
  • Receives a signed JWT / OAuth 2.0 Bearer Token (or AWS SigV4 credentials for AWS-native clients)
  • Attaches the token to every HTTP request in the Authorization header
2
HTTPS Request → AgentCore Gateway Transport

The signed request travels over TLS 1.2+ to the AgentCore Runtime endpoint. AWS PrivateLink keeps traffic inside the VPC — it never traverses the public internet for enterprise deployments.

  • Endpoint: POST /agents/{agentId}/sessions/{sessionId}/text
  • Headers carry: Authorization: Bearer <token> and x-api-key (if API Gateway is fronting)
  • All traffic is logged to AWS CloudTrail for audit
3
AgentCore Identity — Token Verification Verification

AgentCore Identity intercepts the request and verifies the presented credential before any model is invoked.

  • JWT path: Validates signature against IdP's JWKS endpoint, checks iss, aud, exp, and sub claims
  • AWS SigV4 path: IAM verifies the HMAC signature and timestamp (±5 min drift allowed)
  • API Key path: Key is hashed and matched against the stored record in API Gateway
  • On failure → returns 401 Unauthorized immediately; request is dropped
4
IAM Role Evaluation & Permission Check Policy

Once identity is confirmed, AWS IAM evaluates whether this caller is authorised to invoke the specific agent and action.

  • IAM policy must allow bedrock:InvokeAgent and bedrock:InvokeModel on the agent ARN
  • Resource-based policies on the agent control cross-account access
  • SCP (Service Control Policies) at the AWS Organisation level may further restrict what is allowed
  • On failure → returns 403 Forbidden; logged to CloudTrail
5
AgentCore Policy — Guardrails & Content Filters Guardrails

The request payload passes through AgentCore Policy — a deterministic enforcement layer that runs outside the LLM reasoning loop.

  • Natural-language policies compiled to Cedar policy statements are evaluated before model invocation
  • Bedrock Guardrails scan for PII, toxicity, topic restrictions, and grounding checks
  • Session isolation is enforced — each session's context is sandboxed
  • Blocked requests return a policy-violation error with an audit trail
6
Authorised — Agent Execution Begins Success

All checks passed. The AgentCore Runtime proceeds with agent execution: loads Memory context, calls the Foundation Model via Bedrock, and orchestrates any tool calls via the Gateway.

  • Token usage, latency, and session ID are emitted to CloudWatch dashboards
  • OpenTelemetry traces capture the full reasoning trace for debugging
  • Response is streamed back to the caller with source citations if RAG is active

Interview key points — Inbound Authentication

What: Inbound auth is how a client proves its identity to AgentCore before any LLM is touched. The three credential mechanisms are: JWT/OIDC tokens (for human/app SSO), AWS SigV4 (for AWS-native service-to-service calls), and API Keys (for simple integrations via API Gateway). After identity verification, IAM authorization checks what the caller is allowed to do — these are two separate steps (AuthN vs AuthZ). AgentCore Policy then enforces business guardrails deterministically outside the model, making them impossible to prompt-inject around. All events are auditable via CloudTrail + CloudWatch + OpenTelemetry.
